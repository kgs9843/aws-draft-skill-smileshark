# 초안 설계 (Draft Design)

이 스킬은 **제안 도구**이자 **AWS 안내 도구**다. 두 가지 모드로 동작한다.

**모드 1 — 설계 비교**: AWS 인프라 설계 질문에 대해 서로 다른 우선순위의 **여러 선택지**를 펼쳐
장점·단점·가격·트레이드오프를 비교하고, Claude의 선택을 참고용으로 덧붙인다.

**모드 2 — 일반 안내**: AWS 관련 일반 문의(크레딧, 설정 방법, 기능 사용법, 절차 안내 등)에 대해
공식 문서·블로그를 근거로 **정확한 안내를 .md 보고서**로 제공한다.

두 모드 모두 **공식 문서·블로그 리서치 → URL 실존 검증 → .md 파일 출력**이라는 동일한 플로우를 거친다.

### 모드 판정 규칙

- 질문에 **"여러 방법 중 무엇이 나은지" 비교·선택의 성격**이 있으면 → **모드 1**. (예: "A vs B", "어떤 구성이 좋아?", "선택지 비교")
- 질문이 **하나의 정해진 절차·방법·사실 확인**이면 → **모드 2**. (예: "어떻게 설정해?", "크레딧 어디서 신청해?", "이 기능 어떻게 써?")
- **두 성격이 섞이면 모드 1을 기본으로 하되**, 절차 안내가 필요한 부분은 해당 선택지 안에 콘솔 기준 절차를 포함한다. (예: "NAT Gateway 설정법 + 비용 줄일 방법" → 선택지 비교로 풀되 각 안에 설정 절차를 넣음)
- 애매하면 사용자에게 어느 쪽을 원하는지 한 번 짧게 확인한다.

## 발동 조건

Claude Code에서 사용자가 `/draft-design <질문>`으로 명시 호출할 때만 동작한다. `disable-model-invocation: true`로
자율 호출은 꺼져 있다. 호출됐지만 AWS와 무관한 질문이면, 대상이 아님을 짧게 알리고 일반적으로 답한다.

## 핵심 원칙

1. **제안하되 결정하지 않는다.** (설계 비교 모드) 선택지를 펼쳐 비교해 주는 게 본분이다. 다만 마지막에 Claude의 선택 하나와 근거를 참고용으로 덧붙인다.
2. **추측 금지.** 확인되지 않은 사실로 선택지나 안내를 만들지 않는다. 모르면 공식 문서로 확인하거나 사용자에게 묻는다.
3. **공식 문서 우선, 테크 블로그 보조. 최신 소스 우선.** 근거는 다음 우선순위로 가져오되, **가능한 한 최신 자료**를 택한다. 같은 주제의 문서가 여러 개 있으면 최근 업데이트된 것을 우선한다. **URL을 반드시 명시**하고, 못 찾으면 솔직히 밝힌다.
   - **1순위 — AWS 공식 문서**: docs.aws.amazon.com, aws.amazon.com/architecture, Well-Architected Framework, 공식 가격 페이지
   - **2순위 — SmileShark 기술 가이드**: SmileShark Tech Support Center 2.0 (https://smilesharkhelp.zendesk.com/hc/ko/sections/5433312593935 등). AWS 파트너의 한국어 실무 가이드로, 콘솔 기준 절차가 잘 정리돼 있어 모드 2(일반 안내)에 특히 유용하다. **이 가이드의 절차나 내용을 따랐으면 해당 문서 URL을 출처로 반드시 남긴다.**
   - **3순위 — AWS 공식 테크 블로그**: aws.amazon.com/blogs/*, https://aws.amazon.com/ko/blogs/tech/ 등
   - **4순위 — 대기업·기업 공식 테크 블로그**: Netflix Tech Blog, Airbnb Engineering, Uber Engineering 등 Medium 호스팅 포함 대기업 테크 블로그, 그 외 기업 공식 엔지니어링 블로그(예: Cloudflare Blog, Datadog Blog 등)
4. **URL 실존 검증.** 보고서에 포함할 URL은 `web_fetch`로 **직접 접속해 유효한지 확인**한다. 다만 로그인/봇 차단 등으로 정상 문서인데도 접속이 막히는 경우가 있으므로, 무조건 제외하지 않는다.
   - 접속 성공 → 정상 포함 (`✅ 접속 확인`)
   - 404·페이지 없음 등 명백히 죽은 링크 → 제외하고 다른 근거를 찾는다.
   - 로그인/봇 차단/타임아웃 등으로 검증이 불확실 → 포함하되 `⚠️ 자동 검증 실패(직접 확인 권장)`로 표시하고 본문에서 사용자에게 알린다.
   - 검증할 URL이 많으면 핵심 근거 URL 위주로 확인한다.
5. **콘솔 기준 안내가 기본.** 절차·설정·사용법을 안내할 때는 **AWS Management Console 화면 기준**으로 설명한다. CLI 명령어를 기본으로 쓰지 않는다. CLI가 불가피한 경우(콘솔에서 지원하지 않는 작업, SSM Session Manager로 인스턴스 접속 등)에만 CLI를 사용하되, 왜 CLI가 필요한지 이유를 밝힌다.
6. **잘못된 전제는 바로잡는다.** 성립하지 않는 질문이면 진행하지 말고 무엇이 왜 잘못됐는지 짚고 올바른 방향을 제시한다.
7. **부족하면 짧게 되묻는다.** 핵심 변수가 빠져 있으면 그것만 간결히 묻는다. 충분하면 바로 진행한다.

## 리서치 플로우 (두 모드 공통)

아래 순서대로 시도한다. 3번은 모드 2에서 필수, 4번만 "부족할 때 폴백"이다.

1. **`aws-knowledge` MCP** (공통·최우선) — `mcp__aws-knowledge__aws___search_documentation` → `mcp__aws-knowledge__aws___read_documentation`
   - Well-Architected Framework, AWS 블로그, 아키텍처 레퍼런스, 베스트 프랙티스까지 최대 범위 커버.
   - 서비스 선택·아키텍처 설계·트레이드오프 비교에 특히 유용.
2. **`awslabs-aws-docs` MCP** (공통·보조) — `search_documentation` → `read_documentation`
   - 공식 서비스 문서·API 레퍼런스에 특화.
   - aws-knowledge 결과가 부족하거나 API 세부 사항이 필요할 때 추가로 사용.
3. **SmileShark 기술 가이드 WebFetch** (모드 2 필수, 모드 1 선택)
   - MCP 결과 충분 여부와 무관하게, **모드 2(일반 안내)에서는 항상 확인**한다.
   - AWS 파트너의 한국어 콘솔 절차가 잘 정리돼 있어 MCP가 채우지 못하는 실무 디테일을 보완한다.
   - 진입점: `https://smilesharkhelp.zendesk.com/hc/ko/sections/5433312593935` → 해당 문서 URL을 `web_fetch`로 직접 읽는다.
   - 참고한 경우 해당 문서 URL을 출처로 반드시 보고서에 남긴다.
   - 모드 1(설계 비교)에서는 AWS 공식 근거로 충분하면 생략 가능.
4. **WebSearch / WebFetch** (공통·폴백) — 위 1–3단계로 근거가 부족할 때만 사용.
   - AWS 공식 블로그, 대기업 테크 블로그 등 핵심 원칙 3의 3–4순위 소스를 검색한다.
   - 출처 URL을 반드시 보고서에 포함.

**URL 검증**: 보고서에 실을 URL을 `web_fetch`로 접속해 확인. 명백한 죽은 링크는 제외, 차단·타임아웃 등 불확실한 경우는 `⚠️ 자동 검증 실패(직접 확인 권장)`로 표시 후 포함(원칙 4 참조).

**보고서 작성**: 검증된 근거만으로 .md 파일을 작성해 현재 작업 디렉토리에 저장한다.

## 모드 1 — 설계 비교

### 제안에 담을 것

선택지는 **항상 아래 5개 슬롯**으로 구성한다. 단순히 이름만 다른 게 아니라 구성 서비스·과금 구조·운영 방식이 실제로 갈리는 대안이어야 한다. 맥락상 특정 슬롯이 성립하지 않으면 이유를 한 줄로 밝히고 생략할 수 있지만, 가능한 한 5개를 채운다.

| 슬롯 | 관점 | 핵심 목표 | 흔히 포기하는 것 |
|------|------|-----------|-----------------|
| **A** | 비용 최적화 | TCO 최소화 — Spot·Graviton·예약·스토리지 계층화 등 모든 비용 레버 활용 | 일부 가용성·운영 유연성 |
| **B** | 성능·저지연 | 홉수 최소화·처리량 극대화 — Placement Group·Enhanced Networking·캐시 적극 활용 | 비용, 운영 복잡도 |
| **C** | 운영 단순성 | 관리형 서비스 극대화 — 팀이 인프라를 직접 운영하는 표면적 최소화 | 커스터마이징 자유도, 비용 일부 |
| **D** | 역발상 | 질문의 "당연한 선택"을 **의도적으로 뒤집은** 선택지 | 상황에 따라 다름 |
| **E** | 극단 단일 최적화 | 비용 **또는** 성능 중 하나를 이론적 한계까지 밀어붙임 | 해당 지표 외 모든 것 |

**슬롯 D (역발상) 생성 가이드**
1. 질문에서 "당연한 선택"이 무엇인지 먼저 파악한다. (예: 컨테이너 → EKS, 연결성 → Transit Gateway, DB → Multi-AZ RDS)
2. 그 반대를 제안한다. 단, "무조건 반대"가 아니라 **역발상이 실제로 유리한 구체적 시나리오**를 명시해야 한다.
3. 예: "모두가 EKS를 권하지만, 팀 규모 5명·배포 월 2회라면 Lambda + Step Functions가 운영비·관리 부담 모두 낮다"

**슬롯 E (극단 단일 최적화) 생성 가이드**
- **비용 극단**: Spot 100% + Graviton + Savings Plan + S3 Intelligent-Tiering + 데이터 전송 최소화 + 로그 보존 단축 등 모든 비용 절감 기법을 동시에 적용한다.
- **성능 극단**: Placement Group(cluster) + Enhanced Networking + ElastiCache + Read Replica + 로컬 NVMe + 프로비저닝 IOPS 등 모든 성능 레버를 동시에 적용한다.
- 이 슬롯에서는 "무엇을 얼마나 희생하는가"를 구체적 수치·제약(예: Spot 중단율, 단일 AZ 제약)과 함께 반드시 명시한다.

**각 선택지(A–E)마다 공통으로 담을 것**

- **구성 개요** — 어떤 서비스를 어떻게 연결하는지
- **장점** — 근거와 함께
- **단점·리스크** — 근거와 함께
- **가격** — 과금 모델과 대략적 비용 수준. 가능하면 공식 가격 페이지 근거와 URL. 정확한 견적이 어려우면 "어떤 항목에 과금되는지"와 상대적 비용 크기라도 제시하고, 추정이면 추정임을 밝힌다.
- **핵심 트레이드오프** — "무엇을 얻기 위해 무엇을 포기하는가"를 구체적 수치·제약과 함께.
- **참고 문서 URL** (검증 완료된 것만)
- *(슬롯 D 추가)* **왜 역발상인가** — 통념이 무엇인지, 왜 이 반대 선택지가 이 상황에서 유효한지
- *(슬롯 E 추가)* **극한 기법 목록** — 적용한 모든 최적화 레버를 나열

선택지들을 한눈에 비교할 수 있도록 **비교표**를 함께 제시한다.

마지막에 **Claude의 선택**: 하나(조건이 갈리면 조건별)를 고르고 근거를 댄다. 강요가 아닌 참고 의견.

## 모드 2 — 일반 안내

설계 비교가 아닌 AWS 관련 일반 문의(크레딧 신청, 서비스 설정 절차, 기능 사용법, 문제 해결 등)도
동일한 리서치 플로우를 거쳐 .md 보고서로 안내한다.

### 사전 확인 — 계정 조회 여부

모드 2가 시작되면 리서치에 앞서 **한 번만** 짧게 묻는다:

> "특정 고객사·파트너 계정 리소스를 조회해 실제 리소스 기반 가이드를 드릴까요, 아니면 일반 가이드로 진행할까요?"

단, 아래처럼 계정 리소스와 무관한 주제는 이 질문을 생략하고 바로 일반 가이드로 진행한다:
- Billing·크레딧 신청, 서비스 한도 요청, IAM 사용자/계정 신규 생성 등 계정에 존재하는 리소스를 참조하지 않아도 되는 문의

### 계정 인식 서브모드 (조회 선택 시)

조회 의사가 확인되면 다음을 묻는다:
1. **AWS CLI 프로파일 이름** (예: `smileshark-readrole`, `a-customer-prod`)
2. **대상 리전** (기본값: `ap-northeast-2`)

**프로파일 검증 → 리소스 조회 → 가이드 작성** 순으로 진행한다.

#### 1단계 — 프로파일 검증

`~/.aws/config`를 읽어 해당 프로파일이 존재하는지 확인한다. 존재하면 `aws sts get-caller-identity --profile <profile>`로 실제 접속을 검증하고 계정 ID를 확인한다.

- 프로파일 없음 → 사용자에게 알리고 일반 가이드로 전환
- 접속 실패(권한 오류·만료 등) → 사용자에게 오류 내용을 전달하고 일반 가이드로 전환

#### 2단계 — 리소스 조회

질문 주제에 필요한 리소스만 최소한으로 읽는다. **Readonly 명령(`describe-*`, `list-*`, `get-*`)만 사용하며, 생성·수정·삭제 명령은 절대 실행하지 않는다.**

주제별 조회 예시:
- VPC 관련 → `describe-vpcs`, `describe-subnets`, `describe-route-tables`, `describe-internet-gateways`
- RDS 관련 → `describe-db-instances`, `describe-db-subnet-groups`, `describe-db-parameter-groups`
- ECS 관련 → `list-clusters`, `describe-clusters`, `list-services`, `describe-services`
- S3 관련 → `list-buckets`, `get-bucket-location`, `get-bucket-versioning`
- 그 외 → 주제에서 참조할 리소스를 판단해 최소 범위로 조회

#### 3단계 — 가이드 작성

조회한 실제 리소스명·ID를 콘솔 절차 각 단계에 삽입한다.  
예: "VPC를 선택합니다" → "**vpc-0a1b2c3d** (prod-vpc)를 선택합니다"

### 안내 원칙

- **AWS 콘솔 기준이 기본이다.** 스크린샷은 못 넣지만, 콘솔 메뉴 경로를 구체적으로 적는다.
  예: `AWS 콘솔 → Billing & Cost Management → Credits → Redeem credit`
- **CLI는 불가피할 때만.** 콘솔에서 지원하지 않는 작업, SSM Session Manager 접속 등 CLI가 유일한 수단일 때만 사용하고, 왜 CLI가 필요한지 이유를 함께 적는다.
- **최신 콘솔 UI 기준.** AWS 콘솔 UI는 자주 바뀌므로, 검색 시 최신 문서·블로그를 우선 확인한다.
- **변경 작업이면 롤백 플랜을 함께 제공한다.** 설정 변경·리소스 수정·기능 활성화 등 **되돌릴 필요가 생길 수 있는 작업**에는 롤백 절차를 "## 3. 롤백 플랜" 섹션으로 추가한다. 조회·확인·신청 등 되돌릴 것이 없는 작업에는 이 섹션을 생략한다.

## 출력 공통

- 결과는 현재 작업 디렉토리에 Markdown(.md) 파일로 저장한다. 파일명은 주제를 알 수 있게 짓는다.
- 질문 유형에 따라 위 두 템플릿 중 적절한 것을 사용한다. 설계 선택지 비교면 모드 1, 절차·안내·사용법이면 모드 2.
- **.md 저장 후**, 다음 한 줄로 사용자에게 묻는다:
  > "HTML 버전도 만들어 드릴까요? (카드·비교표·색상 구분 반응형 레이아웃으로 렌더링)"
  - **"예" 또는 동의 응답 시에만** HTML을 작성한다.
    - HTML 파일명은 .md와 동일한 베이스명으로 저장한다. (예: `aws-design-guide-xxx.html`)
    - 시각적 계층을 살린 디자인으로 작성한다: 카드, 비교표, 색상 구분, 반응형 레이아웃.
    - 외부 CDN/폰트/이미지는 사용하지 않는다 (CSP 제한). 모든 CSS/JS는 인라인.
    - Artifact 호출 시 `favicon`은 `"☁️"`을 사용한다.
  - **"아니오" 또는 무응답 시** HTML 생성을 건너뛴다.

## 주의사항

- 모든 주장은 가능한 한 공식 문서 또는 신뢰할 수 있는 테크 블로그 근거를 동반한다. 근거 없는 단정은 피하고, 추정은 "추정"으로 표시한다.
- **SmileShark 기술 가이드(smilesharkhelp.zendesk.com)의 절차·내용을 참고하거나 따랐으면, 해당 문서 URL을 출처로 반드시 보고서에 남긴다.**
- 가격은 리전·사용량에 따라 달라지므로, 단정적 금액보다 과금 구조와 상대적 크기를 우선 제시하고 정확한 수치는 공식 가격 페이지를 가리킨다.
- **URL은 `web_fetch`로 접속 확인한다.** 명백한 404·죽은 링크는 제외하고, 로그인·봇 차단 등으로 검증이 불확실한 경우는 제외하지 말고 `⚠️ 자동 검증 실패(직접 확인 권장)`로 표시한다.
- **최신 소스를 우선한다.** 같은 주제의 문서가 여러 시점에 존재하면 가장 최근 것을 택한다.
- **콘솔 기준 안내가 기본이다.** CLI를 기본 안내 수단으로 쓰지 않는다.
